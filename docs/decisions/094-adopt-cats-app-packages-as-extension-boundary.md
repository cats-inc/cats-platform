# ADR-094: Adopt Cats App Packages as the Extension Boundary

## Status

Proposed

## Date

2026-04-29

## Context

ADR-048 separated platform-owned `products` from installable `apps`.

The platform now needs a concrete extension boundary that can carry three
different shapes without inventing three unrelated systems:

1. agent connector capability packs, such as GitHub, Notion, browser, MCP,
   runtime backend, or model-provider integrations
2. first-party product modules, such as a future Cats Learn module that may
   contribute a top-level product surface
3. user-built apps exported from Cats Code, installed into the current host, and
   shown under the Lobby `Apps` section

The reference systems point at complementary lessons:

- OpenClaw is strongest at manifest-first capability declaration, cold registry
  data, targeted activation, and strict SDK boundaries.
- Paperclip is strongest at lifecycle governance, persisted registry state,
  capability-gated worker access, UI contributions, jobs, tools, and admin
  management.

Cats needs the same broad shape, but must keep the public product vocabulary
from ADR-048:

- `product`: platform-owned top-level first-party experience
- `app`: installable and publishable unit
- `plugin` or `extension`: internal implementation vocabulary only

The immediate user-facing driver is the Lobby `Apps` section. It currently has a
mock Pomodoro card. The next architecture must let Cats Code produce a real
installable user app that appears in that section.

## Decision

`cats-platform` will introduce **Cats App Packages** as the installable extension
boundary.

A Cats App Package is a manifest-first bundle that may contribute one or more
host-visible surfaces or capabilities:

- a Lobby app entry under `/apps/:appId/*`
- a first-party product module descriptor under the `Products` section
- agent tools and connector capabilities
- settings sections
- scoped API routes
- background jobs or scheduled tasks
- runtime adapter declarations that are brokered through `cats-runtime`

The public concept is `app`. The host may internally call the loader an
extension host or plugin runtime, but product copy and user-facing inventory must
say `Apps`.

## Package Categories

The manifest must classify each package by extension category:

- `user-app`
  - installed by the owner, usually generated by Cats Code
  - can appear in Lobby `Apps`
  - can expose app routes, app settings, scoped storage, and optional tools
  - is local-trusted in the first implementation

- `capability-connector`
  - provides agent external-link capability
  - may expose auth/setup/settings and agent tools
  - does not need a main app route
  - appears in settings or integrations inventory, and may also add a small
    Lobby entry if it has a meaningful user surface

- `product-module`
  - first-party or system-trusted module
  - may contribute a `PlatformProductDescriptor`
  - may also appear in installed apps inventory as the package that owns the
    product module
  - third-party packages cannot contribute top-level products until a later
    governance decision allows it

## Trust Tiers

The host will treat category and trust separately.

- `system`
  - bundled first-party packages
  - may contribute products and privileged host integration

- `local-user`
  - owner-created packages, especially Cats Code exports
  - may contribute Lobby apps, scoped APIs, settings, jobs, and tools after
    owner approval
  - is not treated as safe for public distribution

- `third-party`
  - future external packages
  - must not be accepted as general executable code until signing, review,
    sandboxing, and update governance are designed

The first implementation should support `system` and `local-user`. It should
represent `third-party` in the schema but keep third-party installation disabled
or gated behind explicit future work.

## Interface Direction

Cats App Packages will use:

- a static manifest named `cats.app.json`
- a typed SDK entry under `@cats-inc/cats-platform/app-sdk`
- manifest validation before runtime activation
- a persisted app registry under the Cats platform data root
- host-owned APIs for install, enable, disable, uninstall, inspect, and launch
- capability-scoped bridge APIs instead of broad direct access to core stores

Runtime-facing extensions must integrate through `cats-runtime` or a
platform-owned adapter layer. Apps must not import `agent-fleet` internals.

## Lobby Integration

The Lobby `Apps` section becomes the first public integration target.

Installed app descriptors must be derived from the app registry and exposed in
the platform host envelope. The hardcoded Pomodoro mock is only a layout preview
and must be replaced by registry-driven installed app entries.

Cats Code exports must be able to produce a valid `user-app` package whose
manifest contributes:

- a visible Lobby app entry
- a route mounted under `/apps/:appId/*`
- optional app settings
- optional scoped API/actions
- optional tools or jobs when the app needs agentic behavior

## Consequences

### Positive

- One extension standard can cover connectors, system product modules, and
  user-built apps.
- The platform preserves the ADR-048 product/app vocabulary instead of leaking
  plugin terminology into user-facing surfaces.
- Cats Code gets a concrete export target: a Cats App Package that the current
  host can install and launch.
- Connector packages can use the same manifest, permission, and lifecycle
  machinery as user apps without pretending to be full UI products.
- First-party modules such as Cats Learn can be packaged as system apps without
  giving third-party packages top-level product privileges.

### Negative

- The app package contract becomes a shared platform API and must be versioned.
- Local user apps are trusted in the first slice; this is not a complete
  third-party security model.
- The host must grow a real app registry and lifecycle layer instead of keeping
  the Lobby `Apps` area as static UI.
- Product modules and apps can be confusing unless documentation keeps their
  terms separate.

### Neutral

- Existing Chat, Work, and Code products do not have to be converted to app
  packages immediately.
- A future Cats Learn module can start as a bundled system app and later become
  optional without changing the package format.
- Public marketplace, signatures, review, and sandboxed third-party UI remain
  later-stage work.

## Alternatives Considered

### Alternative 1: Build separate systems for products, connectors, and user apps

- **Pros**: each domain can be optimized independently
- **Cons**: duplicate manifest, install, settings, permissions, and lifecycle
  mechanics
- **Why not preferred**: the three use cases share enough host integration to
  justify one package standard

### Alternative 2: Use `plugin` as the public concept

- **Pros**: familiar to developers and matches some internal architecture notes
- **Cons**: weaker user-facing language and conflicts with ADR-048
- **Why not preferred**: Cats should present installable user work as apps, not
  as implementation-level plugins

### Alternative 3: Let Cats Code export arbitrary Vite apps only

- **Pros**: easy to build and preview
- **Cons**: no permissions, no registry, no app settings, no agent tools, no
  scoped storage, and no clear Lobby integration
- **Why not preferred**: Cats Code exports need to become first-class installed
  Cats apps, not external web projects loosely linked from the host
