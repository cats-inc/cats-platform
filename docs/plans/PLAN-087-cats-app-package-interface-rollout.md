## Overview

This plan turns the Lobby `Apps` placeholder into a real app-host integration
surface and establishes the standard Cats Code can export to.

The rollout must support three package shapes:

- `user-app` packages generated by Cats Code
- `capability-connector` packages for agent external-link capability
- `product-module` packages for system-trusted first-party products such as a
  future Cats Learn

The first implementation should stay local/self-hosted. Public marketplace,
third-party install, signing, and full sandboxing are explicitly later work.

Inventory notes:

- The hardcoded Lobby Pomodoro card was removed; Lobby `Apps` now reads only
  `PlatformHostEnvelope.installedApps`.
- Existing `PlatformProductDescriptor` remains the Products source. App
  packages extend the host envelope through `PlatformInstalledAppDescriptor`
  instead of replacing product descriptors.
- Cats Code export remains a later phase, but the host side now accepts a local
  package folder containing `cats.app.json`.
- Connector and scoped API declarations are manifest-visible, but execution is
  not mounted yet. Runtime-backed behavior must remain behind platform-owned
  tool/runtime boundaries and `cats-runtime`.
- No app package contract imports `agent-fleet`; that boundary remains outside
  this standard.

## Cats Code Export Acceptance Path

The overnight implementation agent should aim for this first vertical slice:

1. Start from a Pomodoro app template in Cats Code.
2. Export it as a folder containing `cats.app.json`, `package.json`, and
   `dist/renderer/index.js`.
3. Validate it through `POST /api/apps/validate`.
4. Install it through `POST /api/apps/install` after user approval.
5. Confirm it appears under Lobby `Apps`.
6. Click the card and open `/apps/user.pomodoro`.
7. Disable it and confirm it no longer appears as an active Lobby launch entry.

## Technical Decisions

- Public vocabulary is `App`, not `Plugin`.
- The package manifest is `cats.app.json`.
- First implementation supports `system` and `local-user`.
- `third-party` is schema-visible but not generally installable.
- User apps mount under `/apps/:appId/*`.
- Scoped APIs mount under `/api/apps/:appId/*`.
- App registry lives outside Chat/Core state.
- Connectors and runtime adapters go through platform-owned boundaries and
  `cats-runtime`.
- Product module contributions require `trustTier: "system"`.

## Testing Strategy

- **Unit Tests**: manifest validation, permission validation, route collision
  validation, registry state transitions, Lobby entry mapping
- **Integration Tests**: local path install, app list, enable/disable,
  uninstall, scoped API route rejection/acceptance
- **Manual Testing**:
  1. Empty registry shows no mock Pomodoro app.
  2. Fixture Pomodoro installs and appears in Lobby.
  3. Launch opens `/apps/user.pomodoro`.
  4. Disable removes Lobby launch entry.
  5. Connector fixture installs without a Lobby card.
  6. Non-system product contribution is rejected.

## Risks and Mitigations

| Risk | Impact | Mitigation |
|------|--------|------------|
| Local user apps become mistaken for safe third-party apps | High | Keep `third-party` disabled, label local-user trust clearly, and require owner approval |
| App packages import host internals directly | High | Provide `app-sdk` interfaces and add lint/test checks once package split starts |
| Lobby app cards drift back to static mocks | Medium | Make `Apps` render only from installed app descriptors |
| Product modules blur product/app vocabulary | Medium | Keep product contribution system-only and preserve ADR-048 terms |
| Connector packages bypass runtime boundary | High | Require connector tools/adapters to route through platform tool/runtime bridges |
| App registry pollutes Chat/Core dev state | High | Store app registry separately and test with temp roots |
