## Purpose

Capture a product vision for `Cats Code` that is intentionally playful,
memorable, and PR-friendly:

> not `vibe coding`, but `cats coding`

This note defines the idea that real cats can participate in software ideation
through a lightweight playful mode, with their choices turned into prompts,
directions, and generated outputs by the platform.

## Core Thesis

`Cats Code` should eventually include a special mode called `Playground`.

In that mode:

- an iPad or other browser client on the local network opens a full-screen
  playful interface
- fish swim across the screen
- each fish carries or represents a generated topic/idea/need/trait
- a real cat paws at the fish
- the system interprets those touches as bounded selections
- `Cats Code` turns those selections into:
  - an idea direction
  - a playful brief
  - a generated landing page
  - or a strange local app prototype

This is not meant to replace normal coding workflows. It is a playful ideation
surface and a signature product/storytelling moment.

## Why This Matters

This idea is powerful for several reasons:

- it creates a phrase that people will remember: `Cats Coding`
- it makes the product emotionally and visually distinct
- it directly honors the deeper origin of the project
- it produces a demo that is instantly understandable on video
- it connects the companion/emotional side of `Cats` to the builder/platform
  side of `Cats Code`

The important thing is not whether the cat "really codes" in a literal sense.
The important thing is that the system can transform playful cat interactions
into actual software directions and outputs.

## Positioning

This should be framed as:

- a `Cats Code` mode
- a `Playground`
- a playful ideation and PR surface

It should **not** be framed as the main product workflow for building software.

The normal serious loop remains:

- human describes need
- `Work` plans
- `Code` implements

The playground loop is:

- cat explores
- system interprets
- system generates
- human enjoys, iterates, or adopts the output

## Suggested Naming

### Primary phrase

- `Cats Coding`

This is the strongest memorable phrase and can serve as the public hook.

### Product placement

- `Cats Code > Playground`

This keeps it anchored inside the product, rather than feeling like a random
standalone gimmick.

### Useful supporting language

- "Real cats can code, too."
- "Turn cat choices into prototypes."
- "Let your cat help invent software."

## Intended User Experience

### Setup

- the user opens `Cats Code`
- switches to `Playground`
- visits the page from an iPad or tablet over LAN
- the display becomes full-screen and touch-friendly

### Interaction

- fish swim across the screen
- each fish corresponds to one generated idea token or direction
- the cat taps/paws at fish
- the system records which fish were chosen and in what sequence

### Interpretation

The chosen fish act as a playful selection language. They can represent:

- problem domains
- visual styles
- moods
- features
- target users
- weird constraints
- content themes

The system does not need to pretend the cat wrote a requirements doc. It only
needs to turn the selections into a coherent playful brief.

### Output

After enough selections, `Cats Code` can generate:

- a landing page
- a toy local app
- a playful concept website
- a weird experiment
- an "AI imagined" local deployable prototype

## The Fish Metaphor

Fish should not just be random targets.

They should work as a generated choice vocabulary.

Possible design direction:

- each fish carries one visible word or very short phrase
- fish clusters imply theme families
- color and motion can imply meaning or rarity
- some fish can act as "wildcards" or "surprise me" triggers

The important principle is:

- interaction stays fun for the cat
- meaning stays interpretable for the human

## Product Boundaries

### What this mode is

- playful ideation
- a PR/demo signature
- a bridge between companion energy and code generation
- a source of surprising prompts and briefs

### What this mode is not

- the canonical professional app-building workflow
- a replacement for `Work`
- a claim that cats literally understand software architecture
- a reason to overcomplicate the serious builder path

## Relationship to the AI-First App Store Vision

This mode fits naturally into the broader "AI-first app store" direction.

That larger vision says:

- users learn the platform inside `Cats`
- use `Work + Code` to build apps
- install those apps back into the platform

`Cats Coding Playground` is a playful branch of that same idea:

- the platform can generate app directions from unusual inputs
- even a cat's playful choices can become a seed for an installable output

It is not the whole platform story, but it is a striking demonstration of what
an AI-first creation platform feels like.

## First Slice Recommendation

The first slice should stay intentionally small and delightful.

### First slice scope

- full-screen tablet-friendly playground view
- fish-based selection interaction
- simple selection capture and session logging
- generated brief from fish choices
- one lightweight output type, preferably:
  - landing page
  - or toy single-page local app

### First slice non-goals

- full arbitrary multi-file app generation from cat input alone
- deep `Work` integration
- complex local deployment pipelines
- advanced game mechanics
- multi-user multiplayer interaction

### Why keep it small

The first goal is to prove:

- this is fun
- this is legible
- this is memorable
- this can create a real output

That is already enough for a strong PR/demo story.

## System Implications

Even though this is playful, it still touches real product seams.

Likely needs include:

- a dedicated `Cats Code` playground route/surface
- LAN-friendly tablet layout
- a lightweight interaction event model
- mapping from fish selections to generated prompts/briefs
- one output/render path into `Cats Code`
- optional artifact persistence for generated outputs

Later versions could connect to:

- `Work` for turning fish selections into structured tasks
- `Code` preview/deploy pipelines
- platform install flow for generated apps

## Risks

### 1. Becoming pure gimmick

If the mode produces no meaningful output, it becomes a joke and stops being a
product asset.

### 2. Overengineering the first version

If the first slice tries to build a full generative game platform, it will miss
the window for delight and clarity.

### 3. Confusing the main product

If this is presented as the normal way to build apps, it will make `Cats Code`
look unserious.

The right framing is:

- signature playground
- playful ideation mode
- delightful side branch of the serious platform

## Recommendation Summary

1. Officially define `Cats Coding` as a `Cats Code` concept.
2. Implement it as `Cats Code > Playground`, not as the main builder workflow.
3. Optimize the first slice for delight, legibility, and demo value.
4. Turn cat interactions into a coherent brief and one small generated output.
5. Keep the first slice LAN/tablet-friendly and full-screen.
6. Use it as both a genuine playful mode and a high-signal PR surface.
