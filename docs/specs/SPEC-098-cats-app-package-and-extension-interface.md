# SPEC-098: Cats App Package and Extension Interface

## Metadata

| Field | Value |
|-------|-------|
| **Status** | Draft |
| **Owner** | Codex |
| **Reviewer** | User |

## Summary

This spec defines the first Cats App Package standard.

The standard must support three extension shapes with one host integration
model:

- agent external-link capability packages
- first-party product modules such as a future Cats Learn
- user-built apps exported from Cats Code and shown under Lobby `Apps`

The immediate product goal is concrete: replace the current hardcoded Lobby
`Apps: Pomodoro` mock with installed app descriptors produced by the app
registry. Cats Code should be able to export a Pomodoro-style user app as a
package that installs into Cats and launches from Lobby.

## Terminology

- `Cats App Package`: an installable manifest-first bundle.
- `App`: user-facing name for installable packages.
- `Product`: platform-owned first-party top-level experience from ADR-048.
- `Product module`: a trusted app package that contributes a product.
- `Capability connector`: an app package that contributes external-link
  capability, tools, auth/setup, or runtime/backend integration.
- `User app`: an owner-created app package, normally generated by Cats Code.
- `App registry`: persisted host-owned install and lifecycle state.
- `App SDK`: typed package interface exposed by the host.

The renderer and user-facing copy should say `Apps`. Internal code may use
`extension host` or `app runtime`, but should avoid public `plugin` wording.

## Goals

- define one manifest standard that can represent connectors, product modules,
  and user apps
- let Cats Code export installable user apps without hand-written host wiring
- make Lobby `Apps` registry-driven
- keep product modules distinct from general user apps
- require manifest validation before runtime activation
- keep app permissions explicit and capability-scoped
- route runtime/backend integration through Cats platform boundaries and
  `cats-runtime`, not direct `agent-fleet` imports
- keep the first implementation local/self-hosted and private-first

## Non-Goals

- public marketplace
- remote third-party install by default
- app signing, review, or public trust policy
- iframe or process sandbox completeness
- converting Chat, Work, and Code into app packages immediately
- letting third-party packages create top-level `Products` entries
- supporting arbitrary database migrations from apps

## App Categories

### `user-app`

Owner-created package, usually exported by Cats Code.

Allowed first-slice contributions:

- Lobby app entry
- app route under `/apps/:appId/*`
- static renderer bundle
- app settings section
- scoped app data storage
- scoped app API/actions
- optional agent tools, if declared and approved
- optional jobs, if declared and approved

User apps must not contribute `Products` entries in the first slice.

### `capability-connector`

External-link capability package for agents and products.

Examples:

- GitHub connector
- Notion connector
- browser/RPA connector
- MCP server connector
- runtime backend adapter
- model/provider connector
- local database connector

Allowed first-slice contributions:

- auth/setup metadata
- settings UI
- capability declarations
- agent tools
- runtime adapter metadata brokered through `cats-runtime`
- optional status or management UI

Capability connectors do not need a main Lobby app entry unless they expose a
human-facing surface.

### `product-module`

System-trusted first-party module.

Examples:

- Cats Learn
- future first-party domain modules
- bundled optional products

Allowed first-slice contributions:

- optional or required `PlatformProductDescriptor`
- product route prefix, if system-trusted
- product settings section
- app inventory entry pointing to the owning package
- app-level capabilities, jobs, and tools as needed

Only `system` trust may contribute a product descriptor.

## Trust Tiers

| Trust Tier | Source | First-Slice Policy |
|------------|--------|-------------------|
| `system` | bundled first-party package | Can contribute products and privileged integration |
| `local-user` | owner-created local package, normally from Cats Code | Can contribute Lobby apps, scoped APIs, settings, tools, and jobs after approval |
| `third-party` | external package from another publisher | Schema-recognized but installation disabled until later security work |

## Manifest

Every package must include `cats.app.json` at the package root.

The host reads and validates the manifest before loading renderer, server, or
worker code.

```json
{
  "schemaVersion": 1,
  "id": "user.pomodoro",
  "displayName": "Pomodoro",
  "version": "0.1.0",
  "description": "Focus timer with break reminders.",
  "category": "user-app",
  "trustTier": "local-user",
  "publisher": {
    "name": "Local User"
  },
  "compatibility": {
    "catsPlatform": "^0.1.0",
    "appSdk": "1.x"
  },
  "entrypoints": {
    "renderer": "./dist/renderer/index.js",
    "server": "./dist/server/index.js"
  },
  "contributions": {
    "lobbyApps": [
      {
        "id": "main",
        "title": "Pomodoro",
        "subtitle": "Focus timer with break reminders",
        "routePath": "/apps/user.pomodoro",
        "icon": "timer"
      }
    ],
    "settings": [
      {
        "id": "preferences",
        "label": "Pomodoro",
        "path": "/settings/apps/user.pomodoro"
      }
    ],
    "tools": []
  },
  "permissions": [
    "ui.route",
    "storage.appData",
    "settings.app"
  ]
}
```

### Required Manifest Fields

```ts
export type CatsAppManifestSchemaVersion = 1;

export interface CatsAppManifestV1 {
  schemaVersion: 1;

  contributions: CatsAppContributions;

```

Supporting manifest interfaces:

```ts
export interface CatsAppPublisher {
  name: string;
  homepage?: string;
  email?: string;
}

export interface CatsAppCompatibility {
  catsPlatform: string;
  appSdk: string;
}

export interface CatsAppEntrypoints {
  renderer?: string;
  server?: string;
  worker?: string;
}
```

### App ID Rules

- System app IDs should use `cats.<name>` or `cats.product.<name>`.
- Local user app IDs should use `user.<slug>` by default.
- Connector IDs should use `connector.<service>` or a reverse-domain namespace.
- IDs are stable install identities, not display labels.
- Route-safe slugs are derived from IDs but stored explicitly when needed.
- The host must reject app IDs that collide with product route prefixes,
  reserved settings paths, or another installed app.

## Contributions

```ts
export interface CatsAppContributions {
  lobbyApps?: CatsLobbyAppContribution[];
  products?: CatsProductModuleContribution[];
  settings?: CatsAppSettingsContribution[];
  tools?: CatsAgentToolContribution[];
  connectors?: CatsConnectorContribution[];
  apiRoutes?: CatsScopedApiRouteContribution[];
  jobs?: CatsJobContribution[];
}
```

### Lobby Apps

Lobby app contributions create cards in the Lobby `Apps` section.

```ts
export interface CatsLobbyAppContribution {
  id: string;
  title: string;
  subtitle?: string;
  routePath: `/apps/${string}`;
  icon?: string;
  maturity?: 'active' | 'preview';
}
```

Rules:

- `routePath` must be under `/apps/:appId`.
- Apps without a Lobby contribution may still be installed as connectors.
- Lobby entries must be derived from registry state, not hardcoded renderer
  mocks.

### Product Modules

Product module contributions create or extend first-party product inventory.

```ts
export interface CatsProductModuleContribution {
  productId: string;
  productName: string;
  subtitle: string;
  routePrefix: `/${string}`;
  group: 'home' | 'office';
  installPolicy: 'required' | 'optional';
  maturity: 'active' | 'preview';
  settings?: CatsAppSettingsContribution[];
}
```

Rules:

- Only `trustTier: 'system'` may contribute products in the first slice.
- Product modules may also appear in installed apps inventory as the owning
  package.
- Chat, Work, and Code remain existing products until a separate migration
  chooses to package them.

### Settings

```ts
export interface CatsAppSettingsContribution {
  id: string;
  label: string;
  path: `/settings/apps/${string}` | `/settings/${string}`;
}
```

Rules:

- User app settings live under `/settings/apps/:appId`.
- Product module settings may live under their product settings namespace only
  when the package is `system` trusted.
- Settings saves must follow the existing Settings toast convention.
- The host-level `Apps` settings entry appears after product settings entries
  such as `Work` and before `Runtime` in the Settings sidebar.

### Agent Tools

```ts
export interface CatsAgentToolContribution {
  name: `${string}.${string}`;
  title: string;
  description: string;
  inputSchema: Record<string, unknown>;
  outputSchema?: Record<string, unknown>;
  requiresApproval?: boolean;
  runtimeBridge?: 'platform' | 'cats-runtime';
}
```

Rules:

- Tool names are app-namespaced.
- Tool execution must flow through the platform tool boundary.
- Connectors must not expose raw secrets to renderer code.
- Runtime-backed tools route through `cats-runtime` or a platform-owned adapter.

### Connectors

```ts
export interface CatsConnectorContribution {
  id: string;
  service: string;
  auth?: CatsConnectorAuthDeclaration;
  capabilities: string[];
  setupPath?: `/settings/apps/${string}` | `/settings/${string}`;
}
```

Connector examples include GitHub, Notion, browser/RPA, model providers, MCP
servers, and runtime backends.

```ts
export interface CatsConnectorAuthDeclaration {
  kind: 'none' | 'api-key' | 'oauth' | 'local-token' | 'external-cli';
  setupHint?: string;
  secretRefs?: string[];
}
```

### Scoped API Routes

```ts
export interface CatsScopedApiRouteContribution {
  routeKey: string;
  method: 'GET' | 'POST' | 'PUT' | 'PATCH' | 'DELETE';
  path: `/${string}`;
  permission: CatsAppPermission;
}
```

Rules:

- App APIs mount under `/api/apps/:appId/*`.
- App API routes must not shadow core, product, setup, runtime, or settings
  endpoints.
- Request payloads and responses should stay JSON-only in the first slice.

### Jobs

```ts
export interface CatsJobContribution {
  id: string;
  title: string;
  schedule?: string;
  permission: 'jobs.schedule';
}
```

Jobs are declaration-only in the manifest. Runtime scheduling and execution are
owned by the host.

## Permissions

Initial permission vocabulary:

```ts
export type CatsAppPermission =
  | 'ui.route'
  | 'ui.lobby'
  | 'settings.app'
  | 'storage.appData'
  | 'agent.tools.register'
  | 'agent.tools.execute'
  | 'connector.auth'
  | 'runtime.adapter'
  | 'jobs.schedule'
  | 'core.read'
  | 'core.write';
```

Rules:

- Requested permissions are shown before install or enable.
- `core.write` requires explicit user approval and should be avoided by simple
  user apps.
- `runtime.adapter` is only activated through the platform runtime boundary.
- `core.write`, `runtime.adapter`, and connector auth are never granted
  implicitly by a Lobby route.

```ts

  };
```

Renderer apps should use the context bridge instead of directly importing host

## App Registry

The app registry is the durable source of installed app state.

```ts
export type CatsAppInstallState =
  | 'installed'
  | 'enabled'
  | 'disabled'
  | 'error'
  | 'upgrade-pending'
  | 'uninstalled';

export interface CatsInstalledAppRecord {
  id: string;
  manifest: CatsAppManifestV1;
  packagePath: string;
  installState: CatsAppInstallState;
  enabled: boolean;
  installedAt: string;
  updatedAt: string;
  lastError?: string | null;
}
```

First implementation storage:

- registry path: `~/.cats/platform/apps/registry.json`
- package install root: `~/.cats/platform/apps/packages/<appId>/<version>`
- app data root: `~/.cats/platform/apps/data/<appId>`

The exact file names may move behind helpers, but the ownership split must
remain clear.

## Platform Host Envelope

The platform host envelope must add installed app descriptors.

```ts
export interface PlatformInstalledAppDescriptor {
  id: string;
  displayName: string;
  publisher: string;
  version: string;
  category: CatsAppCategory;
  trustTier: CatsAppTrustTier;
  installState: CatsAppInstallState;
  enabled: boolean;
  lobbyEntries: CatsLobbyAppContribution[];
  settings?: CatsAppSettingsContribution[];
}
```

Lobby reads this descriptor list to render `Apps`.

## Cats Code Export Standard

Cats Code should export user apps in a host-installable folder or archive.

### Exported Folder Shape

```
my-pomodoro.cats-app/
  cats.app.json
  package.json
  README.md
  dist/
    renderer/
      index.js
      assets/
    server/
      index.js
```

### Cats Code Export Requirements

Cats Code export must:

1. generate `cats.app.json`
2. set `category: "user-app"` and `trustTier: "local-user"`
3. include at least one `lobbyApps` contribution for visible apps
4. build renderer assets into `dist/renderer`
5. build optional server/action code into `dist/server`
6. run manifest validation before offering install
7. show requested permissions before install
8. call the platform install API after explicit user approval
9. open the installed app route after successful install when requested

### Install Handoff

The first install handoff should be local and explicit:

1. Cats Code creates the package in a workspace export directory.
2. Cats Code validates the package through the host app validation API.
3. Cats shows a review screen with identity, route, and permissions.
4. User approves install.
5. Host copies or links the package into the app install root.
6. Host updates the app registry.
7. Lobby refreshes from the platform envelope.
8. The new app appears under `Apps` and can be launched.

## Lifecycle

First lifecycle commands:

- `validate`
- `install`
- `enable`
- `disable`
- `uninstall`
- `inspect`
- `launch`

Later lifecycle commands:

- `upgrade`
- `approve capability escalation`
- `export`
- `publish`
- `sign`

## API Surface

Planned host APIs:

- `GET /api/apps`
- `GET /api/apps/:appId`
- `POST /api/apps/validate`
- `POST /api/apps/install`
- `POST /api/apps/:appId/enable`
- `POST /api/apps/:appId/disable`
- `DELETE /api/apps/:appId`
- `POST /api/apps/:appId/actions/:actionKey`
- `ALL /api/apps/:appId/scoped/*`

All mutation endpoints require owner approval in the UI. Agents must not write
demo app records into user state without explicit approval, following the state
hygiene policy in [AGENTS.md](../../AGENTS.md).

## Security Model for First Slice

The first slice is local and private-first.

Allowed:

- bundled system apps
- local user apps generated by Cats Code
- owner-approved local path installation

Not allowed by default:

- remote third-party package install
- public marketplace
- unreviewed runtime adapter activation
- app route access to raw secrets
- app direct imports from core store internals

The first implementation may use trusted same-origin renderer loading for local
user apps, but the manifest and SDK should leave room for a later sandboxed
renderer.

## UX Requirements

- Lobby `Apps` renders installed app entries from the registry.
- Empty or unavailable apps show a quiet empty state, not a mock Pomodoro card.
- A Cats Code exported app appears with its display name, subtitle, publisher,
  and preview/active maturity.
- App lifecycle management lives in Settings under `Apps`, positioned under
  `Work` and above `Runtime`.
- Apps with no main UI should be listed in Settings or Integrations, not forced
  into Lobby.
- The install review screen must show permissions in plain language.
- Disable/uninstall must remove the app from active Lobby entries without
  deleting app data unless the user chooses purge.

## Acceptance Criteria

- A local Pomodoro app package generated from Cats Code can validate.
- The same package can install into the local host after user approval.
- The package appears under Lobby `Apps`.
- Clicking the Lobby card opens `/apps/user.pomodoro`.
- The app can read/write only its scoped app data.
- Disabling the app removes the active Lobby launch entry.
- A capability connector can install without a Lobby card and still expose
  declared tools/settings.
- A system product module can contribute a product descriptor only when marked
  `trustTier: "system"`.

## Open Questions

- Should first-slice renderer loading use direct ESM import, iframe isolation,
  or a host-owned component adapter?
- Should app packages be copied into the install root or symlinked during local
  development?
- Should local-user app IDs be globally unique or only unique per Cats data root?
- What minimal app component kit should be exposed in `app-sdk` without locking
  user apps to the full internal design system?
- Should Cats Code export a `.catsapp` archive immediately or start with
  installable folders?
