# Terminology

> Short definitions used in this project.

## Standards and Protocols

| Term | Meaning |

## Interaction Core Terms

Canonical naming rule:

- `Conversation` is the canonical shared term for the durable interaction unit
  across `Chat`, `Work`, and `Code`.
- `thread` is allowed as UI copy or informal shorthand, but it should not
  replace `Conversation` in shared schemas, APIs, or cross-product contracts.
- Names such as `chat_thread`, `code_thread`, and `work_thread` should be
  treated as kinds of `Conversation`, not as separate top-level record types.

| Term | Meaning |
|------|---------|
| Agent | A canonical Interaction Core record (`AgentRecord` / `CoreActorRecord`). Reusable identity for an owner, orchestrator, worker, stakeholder, bot, or resource. `Cat` is the product-facing projection over an Agent. See the full product-facing definition under **Product Terms** (`Entity` / `Agent`). |
| Participant | A canonical Interaction Core record (`ParticipantRecord`). One `Agent`'s durable membership inside one `Conversation`, carrying `role`, `status`, `joinedAt`, and `metadata` — nothing more on the canonical record. Execution-lease / runtime-attachment state lives on `Session`, not on Participant; product projections may compose the two, but the canonical record does not carry lease fields. See the product-facing definition under **Product Terms**. |
| Transport binding | A canonical Interaction Core record (`TransportBindingRecord`). Represents an external transport thread (e.g. a Telegram DM) and its optional links into Cats — `conversationId`, `participantId`, and `agentId` are all nullable, so a binding may resolve to a `Conversation` or remain unresolved (e.g. ingress-only, or pending assignment). Lives in Interaction Core because when it does resolve, it is what selects the `Conversation` a transport ingress lands in. See the product-facing definition under **Product Terms**. |
| Container | An optional parent grouping that owns one or more conversations, such as a parallel-comparison surface. |
| Conversation | One durable interaction boundary with its own transcript and execution state. `Conversation` is the canonical Core/API/schema term for the durable thread-like unit across `Chat`, `Work`, and `Code`. |
| Thread | A UI-facing or informal alias for a `Conversation` or product-local room. Use `thread` in copy when it reads more naturally, but do not promote it over `Conversation` in shared technical contracts. |
| Turn | One user- or system-initiated dispatch cycle inside a conversation. |
| Lane | One stable target-specific response track inside a turn. Lane identity is durable and must not be conflated with runtime session identity. |
| Segment | One lane-local product-visible unit such as text, tool, or status delivery. A segment is product-normalized and is not required to equal a provider-native block or chunk. |
| Session | One runtime attachment generation for a lane. `sessionId` is ephemeral runtime identity, not durable lane identity. |
| Scheduler policy | The rule that decides whether lanes activate serially or concurrently. |
| Sharing policy | The rule that decides whether execution happens in one shared conversation, isolated child conversations, or another bounded context-sharing model. |
| Coordinator capability | An optional capability layer, such as `Boss Cat`, that may influence routing or orchestration without redefining the core interaction model. |
| Concurrent response cluster | The product projection for one concurrent turn inside one conversation. It groups many stable lanes under one user-turn fan-out. |
| Parallel container | A parent grouping that owns many child conversations, such as `Parallel Chat` or future `Peer code`. It is not the same thing as a concurrent response cluster. |
| Materialization | The process by which interaction outcomes become durable structured product state outside the transcript projection. |
| Mutation | A structured output proposing or applying a change to durable product state. |
| Artifact | A Run / Task by-product record; see canonical definition under **Managed Work and Execution Terms**. |
| Reference | A structured pointer field shape (not a record family); see canonical definition under **Managed Work and Execution Terms**. |
| Projection | A product-specific view over shared canonical state, such as a chat transcript, work dashboard, or code review pane. |
| Lens | A stable view over one canonical product home or record family. In `MY CATS`, lenses such as `Overview`, `Chat`, `Work`, and `Code` expose different agent state without forking identity. |
| Contextual subset | A product-local subset or panel that shows only the state relevant to the current surface, such as `Cats in this chat`, assigned agents in Work, or a Code crew list. A contextual subset is not a second registry and should deep-link back to the canonical platform home when appropriate. |
| Runtime capability profile | The normalized description of how much delivery richness a runtime/backend exposes, such as rich streaming, text streaming, or terminal-only result delivery. |
| Provider capability bootstrap config | The operator-owned YAML registry that gives an initial `default`, `strong_agent`, or `weak_worker` treatment to explicitly listed provider/model/control targets. Targets not listed in this config start default/unknown, regardless of provider name or runtime delivery richness. |
| Provider capability profile | The policy input describing observed or configured model ability such as tool-use accuracy, JSON fidelity, reasoning depth, context reliability, and recovery reliability. It is separate from the runtime capability profile and is refined by bootstrap config, eval evidence, session history, and operator overrides. |
| Normalized delivery event | A product-owned runtime event used by transcript, repair, replay, and materialization logic after adapter-specific payloads have been normalized. |
| Guide Cat assist capability | The optional low-privilege assist layer that may generate greetings, prompt chips, helper copy, and contextual suggestions for setup, lobby, chat entry, or other surfaces, while degrading cleanly into deterministic fallback when unavailable. |
| Guide Cat assist bundle | One locally persisted snapshot of Guide Cat assist content for a normalized surface scope. A bundle may include greeting copy, chips, recap, feature guidance, provenance, and freshness metadata. |
| Deterministic assist baseline | The built-in surface-specific fallback content used when no fresh or cached Guide Cat assist bundle is available. |
| Execution profile | A durable preset or binding that captures runtime-affecting inputs such as `cwd`, worktree mode, permission profile, tool/skill profile, and memory profile for a participant, lane, or child conversation. |
| New Chat | The `Cats Chat` entry preset. Conversation-first: it creates or resumes the Interaction Core records needed for a chat, especially a `Conversation`, and does not force a `Task`, `Run`, `Project`, or `WorkItem` at creation time. |
| New Code | The one-person `Cats Code` entry preset. Task-first conversation: it creates one primary coding `Conversation` and one primary Code-owned `Task`, but does not force a `Project`, `WorkItem`, or `Run` at creation time. If first send immediately dispatches a coding agent, the first `Run` may be emitted in that submit flow by the execution dispatcher. |
| Team Code | The shared-room `Cats Code` entry preset that creates one multi-participant coding conversation with workflow policy. Code presets remain Code-owned and task-first, but runs are created only when execution attempts are admitted by the dispatcher/runtime bridge. |
| Peer Code | The branch/review `Cats Code` entry preset that creates one parallel container with many child coding conversations and optional automation policies. Branch/review runs are attempt records, not entry records. |
| New Work | The `Cats Work` entry preset. Managed-work-first conversation: it creates one primary `Conversation`, one `Project`, one `WorkItem`, and one primary `Task` linked through `WorkItem.taskId`; a `Run` is created later when supervised execution actually starts. |
| Task product binding | The derived Work Graph task projection label: `work`, `code`, `chat`, or `unbound`. Precedence (encoded by `resolveTaskProductBinding` in `src/products/work/api/projectionSupport.ts`): (1) `WorkItem.taskId` bridge → `work`; (2) `build` / `preview` `Artifact` on the task → `code`; (3) explicit planning provenance (`planning.productHint` or `planning.transfer.suggestedProduct`) → that product, with explicit `'work'` without a WorkItem demoted to `unbound` and surfaced as a `missing_planning_execution_bridge` diagnostic via `detectIncompleteWorkClaim` + `buildWorkGraphProjection`; (4) `Conversation.kind === 'code_thread'` legacy fallback → `code`. Chat-* / `work_thread` / `direct_message` / `external_transport` / `private_escalation` conversation kinds alone do **not** bind: chat requires explicit chat planning provenance per the deliberate-only Chat Task producer rule, and work requires the structural `WorkItem.taskId` bridge. The projection exposes only the *current* binding; if a Code-origin task is later linked through `WorkItem.taskId`, the projection emits `work`, and every Code-origin signal that drove the pre-promotion binding still persists on the underlying records (`planning` metadata for hint-driven origin, `Conversation.kind = 'code_thread'` for conversation fallback, attached `build` / `preview` `Artifact` records for artifact-driven origin) for diagnostic / audit. None of these is rewritten by promotion, but none is surfaced as a separate field on `WorkGraphObjectSummary`. |
| No project task bucket | The honest Work UI home for tasks with no Project / WorkItem lineage. It may be grouped or labeled by task product binding, especially `code`, `chat`, or `unbound`, and must not silently create a fallback project. There is no `No project / work` subgroup because `work` requires the Planning bridge. |
| Task family | A Code preset grouping made of one primary/root `Task` plus child tasks connected by `Task.parentTaskId` for explicit sub-objectives. It is not a separate Core record, not a parallel container, and not a Work `Project` / `WorkItem` substitute. |
| Code Workspace | A Code-owned execution-context projection for repo/folder/worktree or managed-room state. Current persisted signals include Code task metadata under `codeWorkspace`, `Conversation.repoPath`, execution-profile `cwd` / worktree policy, and run/runtime metadata. It is not a `Project`, not a `WorkItem`, and not a Core record family. It may back the Code `Workspaces` sidebar and must not create Work Planning anchors by itself. |
| Code Artifacts sidebar | The Code-owned navigation entry that indexes Code-relevant `CoreArtifactRecord` rows by task, run, conversation, workspace, kind, and status. It is a sidebar/read-model surface, not a separate artifact record type. Work may show the same artifact as anchored evidence, but Code owns the `Artifacts` sidebar experience. |
| Artifact declaration | A structured producer request for Cats Code to record or propose an artifact. Valid producers are agent, tool, system candidate detection, and user import. The declaration is not itself a Core record and is not trusted directly; Cats Code validates anchors, paths, kind, status, idempotency key, publish-action context, and provenance before writing `CoreArtifactRecord`. Phase 1 agent declarations travel through the Cats-native Code runtime action `declare_artifact`; transcript prose and JSON-looking chat snippets are not artifact declarations. |
| Candidate artifact | An artifact declaration that Cats Code has not promoted to ready/published output. The first implementation may represent it as a draft `CoreArtifactRecord` with candidate metadata or hold it in a Code-local pending queue; either way it is distinct from a source-file edit. |

## Managed Work and Execution Terms

Canonical taxonomy (see [ADR-081](./decisions/081-canonicalize-three-tier-core-record-taxonomy.md)). The authoritative record set is declared in `CORE_CANONICAL_RECORD_FAMILIES` (`src/core/types.ts:21-34`). This layering groups those families; it does not replace them.

- **Planning layer** (independent entities): `Project`, `WorkItem`.
- **Execution layer** (independent entities): `Task`, `Run`, `Mission` (distinct record, not a `Task` subtype — see ADR-063; all its anchoring fields, including `managedWorkId`, are nullable, so a mission may or may not tie to a `WorkItem`).
- **Cross-cutting approval gate** (not a layer): `CoreApprovalRecord` is embedded on `CoreTaskRecord.approval`; `CoreApprovalBindingRecord` is the independent record at `core.approvalBindings` and attaches approval onto any subject (`project | work_item | task | run | artifact | conversation`).
- **Materialization records** (durable state produced around the Planning / Execution / Interaction graph, not peer entities of `Run`): `Artifact`, `Outcome`, `Checkpoint`, `Trace`, `Activity`. This tier uses **record-specific nullable anchors**, not a uniform set — `Trace`, `Checkpoint`, and `Outcome` carry only `conversationId` / `runId` / `taskId` and cannot anchor directly at the Planning layer, while `Artifact` and `Activity` additionally carry `projectId` / `workItemId` and may attach to a project / work item / conversation with no `Task` or `Run`. Some rows are run-bound execution evidence when `runId` is present (e.g. `Outcome`, `Checkpoint`, `Trace`, `build` / `preview` `Artifact`); others have no `Run` at all (e.g. operator-authored `Activity` with kinds `note` / `operator_action` / `work_item_updated`, or `Artifact` of kind `document` / `report` / `attachment` / `transcript_export` / `dataset`). `Activity` is an independent operator-feed record with its own `/api/core/activities` surface — it is **not** a projection over `Trace`, and consumers must read it directly.
- **Rules, not entities**: `Schedule`, `Trigger`, scheduler / sharing / dispatch / delivery / budget policies.

Surface-level concepts such as `Goal`, `Requirement`, `Backlog Item`, `Issue`, `Story`, `Epic`, `Defect` are **`WorkItem` kinds**, not separate record families. `Work Task`, `Code Task`, and `Chat Task` are all `Task`. `Execution Result` is `Outcome`. `Assignment` is a UI synonym for `Mission`.

Note: `AgentRecord`, `ParticipantRecord`, and `TransportBindingRecord` are also canonical record families per `CORE_CANONICAL_RECORD_FAMILIES`; they sit in the Interaction Core layer and have their own rows in the **Interaction Core Terms** table above.

| Term | Meaning |
|------|---------|
| Managed Work | The operator-facing Planning layer. Independent durable records: `Project`, `WorkItem`. Surface-level concepts like goals, requirements, backlog items, and issues are `WorkItem` kinds, not separate records. `Approval` is not listed here because it is cross-cutting — see the `Approval` row below. |
| Project | The top-level Planning container. Holds `WorkItem` rows and links to a primary `Conversation`. Not to be confused with the platform-level `project_workspace` `Container` kind. |
| Work Item | The generic Planning unit (`CoreWorkItemRecord` / `ManagedWorkRecord`). Self-nestable via `parentWorkItemId`. Bridges into Execution via the optional `taskId` link. Subsumes goals, requirements, backlog items, issues, stories, epics, and defects via metadata kind. |
| Work Task | Legacy alias for a `Task` that lives on the Work side. In shared contracts prefer `Task`. |
| Code Task | A `Task` that the Code product owns. Product routing priority: (1) presence of a `build` or `preview` `Artifact` on the task forces Code; (2) otherwise `resolveTaskExecutionProduct` consults the task's planning handoff — `planning.productHint === 'code'` or `planning.transfer.suggestedProduct === 'code'` is authoritative; (3) the linked `Conversation.kind === 'code_thread'` is only a legacy / no-planning fallback. Not a separate record type — always a `CoreTaskRecord`. |
| Chat Task | A `Task` that Chat deliberately materializes from an already-active Chat conversation, such as a user-created follow-up / action item or assistant-generated chat-side plan. It is not created by default for `+New chat`, not created by Chat -> Code or Chat -> Work draft activation, and not a separate record type. It does not require a Project or WorkItem unless explicitly linked into Work. Work Graph surfaces should show it with `productBinding = chat` and place it under `No project` when no Planning lineage exists. The deliberate-only producer rule is enforced by the projection: a chat-* / DM / external-transport / private-escalation conversation alone does **not** make `productBinding = chat`; explicit `planning.productHint = 'chat'` or `planning.transfer.suggestedProduct = 'chat'` is required, otherwise the task projects as `unbound`. |
| Task | The Execution orchestration unit (`CoreTaskRecord`). Self-nestable via `parentTaskId`. Carries approval state inline via `CoreTaskRecord.approval` (a `CoreApprovalRecord` value object, not a standalone table). Parent of `Run`. A task may be Work-, Code-, Chat-, or unbound-attributed at projection time. |
| Mission | A distinct Execution-layer record (`MissionRecord`), not a `Task` variant — ADR-063 deliberately separated the two. All anchoring fields are nullable: `managedWorkId` (optional `WorkItem` anchor), `assignedAgentId` (optional agent binding), `conversationId` / `sourceTurnId` / `sourceLaneId` (optional interaction provenance). One `WorkItem` may spawn many missions, and some missions remain fully internal and never surface as `WorkItem` rows. Do not document missions as requiring a `WorkItem` anchor. |
| Assignment | UI-facing synonym for `Mission`. Do not use in shared schemas. |
| Approval | A cross-cutting gate, not a layer. `CoreApprovalRecord` is embedded on `CoreTaskRecord.approval` and captures the approval state of a single task; it is not a standalone table. `CoreApprovalBindingRecord` is the independent record (persisted at `core.approvalBindings`) that attaches an approval task onto any subject whose `subjectKind` is `project | work_item | task | run | artifact | conversation`. |
| Run | One concrete execution attempt for a `Task` or `Mission` (one coder session, tool batch, build, retry, takeover). Self-nestable via `parentRunId`. A run is emitted when the execution dispatcher / runtime bridge admits an executable attempt, not when an entry shell or Task is created. One task accumulates many runs over time. |
| Execution dispatcher / runtime bridge | The platform seam that admits an executable attempt for a `Task` or `Mission` by writing a `CoreRunRecord`. The canonical primitive is `upsertCoreRun` in `src/core/model/executionRecords.ts`; a `Run` exists from the moment that primitive successfully writes, regardless of whether the downstream runtime call (if any) succeeds. The runtime-client calls (`RuntimeClient.createSession` / `sendMessage` / `createWakeup` in `src/runtime/client.ts`) are **not** the admission boundary: every `Run` materialization happens through a *synchronous* `upsertCoreRun` write somewhere in the call chain, either before the runtime call (Code, Work, supervision, checkout paths) or as a separate downstream sync call (assignment-driven path described below). Principal dispatcher call sites: (1) **synchronous start paths that write the Run before any runtime call** — `src/core/taskLifecycle.ts:checkoutTaskExecution` (actor checks out an approved task; writes a `running` Run; invoked by `POST /api/core/tasks/:id/checkout` and by per-product checkout helpers), `src/products/code/state/taskExecution.ts:bridgeCodeTaskToRuntime` (Code interactive execute; writes a `running` Run before `createSupervisedRuntimeSession`), `src/products/work/api/index.ts:launchWorkSupervisedRun` (Work supervised-run launcher; writes a `queued` Run before runtime takeoff). (2) **runtime-feedback watcher** — `src/core/taskLifecycleWatchers.ts:startTaskRunWatcher` reconciles state on an *existing* `Run` identified by the `runId` the caller passes in; `reconcileObservedTaskRun` returns early if no `Run` with that id exists in Core (`taskLifecycleWatchers.ts:71-73`). The watcher therefore **does not bootstrap the first `Run`**; it only updates state on a `Run` that some sync path has already written. (3) **supervision / operator / external-recording paths** — `src/platform/supervision/lifecycleTools.ts`, `src/platform/supervision/approvalRequests.ts`, `src/core/api/controlOperatorActions.ts`, `src/core/api/recordExecutionRoutes.ts`. Note on assignment-driven dispatch: `src/core/taskLifecycle.ts:applyTaskAssignmentLifecycle` calls `RuntimeClient.createWakeup` (with payload formed by `buildTaskRuntimeExecutionRequest` in `src/shared/taskExecutionBridge.ts`) but does **not** itself call `upsertCoreRun` and the watcher cannot bootstrap from a wakeup either. The first `Run` for that path therefore materializes only when something downstream of the wakeup invokes a sync `checkoutTaskExecution` — typically the runtime/agent calling `POST /api/core/tasks/:id/checkout` once it has a session id to associate with the assignment. Without that downstream sync write the wakeup is queued but no `Run` is created. The Code first-send case typically admits within the same submit flow as the entry, so `+New code` Run-laziness is on the order of seconds; Work usually waits for a separate supervised execute / continue action, so its laziness window is much longer. New producers shall route Run creation through `upsertCoreRun` rather than re-implementing parallel persistence. |
| Outcome | The durable success/failure conclusion for a `Task` or `Run` (`CoreOrchestrationOutcomeRecord`). Execution evidence — usually run-bound, but `conversationId`, `runId`, and `taskId` are all nullable, so an outcome may be run-bound, task/conversation-bound, or temporarily unanchored. Not a peer of `Run`. Legacy terms `Execution Result` / `Result` resolve here. |
| Checkpoint | A mid-execution state snapshot (`CoreCheckpointRecord`). Execution evidence — usually run-bound, but `conversationId`, `runId`, and `taskId` are all nullable, so a checkpoint may be run-bound, task/conversation-bound, or temporarily unanchored. Not a peer of `Run`. |
| Trace | The execution event log (`CoreTraceRecord`) produced during Task / Run execution — `note`, `status`, `dispatch`, `approval`, `checkpoint`, `outcome`, `error` kinds. Execution evidence — usually run-bound, but `conversationId`, `runId`, and `taskId` are all nullable, so a trace may be run-bound, task/conversation-bound, or temporarily unanchored. A distinct record from `Activity`. |
| Activity | An independent operator-feed record (`CoreActivityRecord`), persisted at `core.activities` with its own `/api/core/activities` GET/POST surface. Its `kind` enum covers `note`, `status_change`, `approval_requested`, `approval_decided`, `operator_action`, `artifact_recorded`, `checkpoint_recorded`, `work_item_updated`. **Not** a projection over `Trace` — operator-authored activities are written straight to this table and do not flow through `Trace`; consumers must read `CoreActivityRecord` directly. |
| Artifact | A durable output (`CoreArtifactRecord`) whose `kind` is one of `document`, `report`, `build`, `preview`, `attachment`, `transcript_export`, `dataset`. A materialization record rather than strictly a Run by-product: `build` and `preview` typically carry a `runId`, but `document` / `report` / `attachment` / `transcript_export` / `dataset` often have no `runId` and attach directly to a project, work item, task, or conversation. All back-references (`projectId` / `workItemId` / `conversationId` / `taskId` / `runId`) are nullable. Opening a Code `Artifacts` sidebar entry does not create an Artifact; Code artifacts are materialized from structured declarations by agents, tools, system candidate detection, or user imports after Cats Code validates and normalizes them. Agent declarations use the Cats-native Code runtime action `declare_artifact` in Phase 1, and system detection is candidate-only. Cwd scanning and transcript JSON parsing are not authoritative artifact producer paths. |
| Schedule | A rule (not an entity) that can create or activate missions / runs later, such as a cron schedule or recurring automation rule. Persisted as configuration, not listed alongside `Task` / `Run` as a peer record. |
| Trigger | The immediate event that starts work — cron tick, webhook, transport ingress, owner click, workflow continuation. A rule-firing event, not a durable entity. |
| Reference | A structured pointer type linking interaction or execution state to files, workspaces, repos, conversations, or other resources. A field shape, not a record family of its own. |
| Job | Overloaded legacy term. In new Cats docs and contracts prefer `Mission` (delegated work) or `Run` (concrete execution attempt) unless an external system explicitly requires the word `job`. |
